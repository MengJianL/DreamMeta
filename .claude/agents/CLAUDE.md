# Meta-Department 元部门架构索引

> Meta-Department 以基础层维持连续性，以编排层建立组织秩序，以执行层处理对象与能力，通过评估—核验闭环保障质量，通过创造—记忆—路由闭环实现自我扩展。

---

## 一、总纲

本元部门由 **13 个不可拆分的原子元**组成，分为三层引力结构。
13 原子不是 13 个并列动作，而是一个从**持续存在 → 组织推进 → 独立治理 → 结果形成 → 能力扩展**的完整结构。

它们共同回答五类核心问题：

1. **系统如何持续存在** — M01 / M02 / M03
2. **系统如何把问题组织起来** — M04 / M05 / M08
3. **系统如何判断做得是否足够好、是否成立** — M06 / M12
4. **系统如何真正产出与推进结果** — M09 / M10 / M11 / M07
5. **系统如何扩展自身能力边界** — M13

---

## 二、三层引力结构

### Layer 1 — Foundation / 基础层

> 提供系统得以存在的三个连续性条件。在所有层运行期间持续提供底座约束。

| 编号 | 原子 | 文件 | 连续性 | 核心职责 |
|------|------|------|--------|---------|
| M01 | memory | `M01-memory.md` | 时间连续性 | 沉淀并治理组织记忆的连续性基础设施 |
| M02 | identity | `M02-identity.md` | 存在连续性 | 定义并维持实体身份与职责边界的治理锚点 |
| M03 | channel | `M03-channel.md` | 交互连续性 | 管理系统与外界交互边界与通道秩序的基础设施 |

基础层闭环：不失忆（M01）· 不失我（M02）· 不断联（M03）

### Layer 2 — Orchestration / 编排层

> 把输入转化为可治理、可推进、可裁决、可交付的组织过程。是系统的"组织中枢"。

| 编号 | 原子 | 文件 | 组织操作 | 核心职责 |
|------|------|------|---------|---------|
| M04 | decompose | `M04-decompose.md` | 结构化 | 将整体目标转化为可治理任务结构的拆解原子 |
| M05 | route | `M05-route.md` | 归属化 | 将任务需求映射到合适执行能力的归属原子 |
| M06 | evaluate | `M06-evaluate.md` | 裁决化 | 对对象质量与达标性进行独立裁决的评估原子 |
| M07 | synthesize | `M07-synthesize.md` | 复合化 | 将多个结果整合为可追溯复合结果的综合原子 |
| M08 | sequence | `M08-sequence.md` | 秩序化 | 为任务结构建立推进秩序的时序编排原子 |

### Layer 3 — Execution / 执行层

> 对已组织好的需求进行具体行动。五种本质不同的执行方式，按对象来源区分。

| 编号 | 原子 | 文件 | 面向对象 | 核心职责 |
|------|------|------|---------|---------|
| M09 | compose | `M09-compose.md` | 新候选内容 | 在约束下构成候选内容对象的生成原子 |
| M10 | retrieve | `M10-retrieve.md` | 既有内容对象 | 从既有来源召回对象、模式与依据的检索原子 |
| M11 | invoke | `M11-invoke.md` | 既有能力对象 | 将任务需求接合到现成能力上的调用原子 |
| M12 | verify | `M12-verify.md` | 对象成立性 | 检查对象成立性与符合性的核验原子 |
| M13 | create | `M13-create.md` | 新能力组织对象 | 将稳定需求转化为新可复用能力组织的创造原子 |

## 三、五类关系网络

### 1. 支撑关系（垂直）

- **Layer 1 → Layer 2**：记忆锚点、身份边界、交互入口。没有基础层，编排层会变成短时临时组织。
- **Layer 2 → Layer 3**：任务结构、执行归属、推进秩序、结果裁决接口。没有编排层，执行层会退化为散乱动作。
- **Layer 3 → Layer 2 / Layer 1**（反向反馈）：执行层产物进入 M06 评估、M07 综合、M01 沉淀、M13 扩展。

### 2. 交接关系（横向）

每个原子把输出显式组织为下一个原子可接手的对象：

- M04 输出任务结构 → M05 / M08 接手
- M05 输出执行归属 → 执行层接手
- M09 / M10 / M11 输出对象 → M12 / M06 / M07 接手
- M12 / M06 输出裁决结果 → M08 / M07 / M03 接手
- M13 输出新对象 → M12 / M06 / M05 / M01 接手

### 3. 制衡关系（对偶）

| 制衡对 | 张力 |
|--------|------|
| M09 ↔ M10 | 新构成 vs 已存在召回 |
| M10 ↔ M11 | 召回既有对象 vs 激活既有能力 |
| M06 ↔ M12 | 质量裁决 vs 成立核验 |
| M04 ↔ M08 | 任务结构 vs 推进秩序 |
| M09 ↔ M13 | 候选内容构成 vs 能力组织创造 |

### 4. 回流关系

- 评估不通过 → 回流 M04 / M05 / M08 / M09 / M13
- 核验失败 → 回流 M09 / M10 / M11 / M13
- 路由失配 → 回流 M05
- 编排失衡 → 回流 M08
- 结构不可治理 → 回流 M04
- 能力缺口稳定存在 → 升级 M13

### 5. 演化关系

执行经验 → 记忆沉淀（M01）→ 反复缺口 → 能力候选（M05/M10）→ 有效模式 → 创造固化（M13）→ 核验与评估（M12/M06）→ 纳入体系路由使用（M05）→ 再沉淀回 M01

## 四、五条核心系统链路

### 主链 1：标准任务治理链

`M03 → M04 → M05 → M08 → M09/M10/M11 → M12 → M06 → M07 → M03`

接入 → 拆解 → 路由 → 编排 → 执行 → 核验 → 评估 → 综合 → 交付

### 主链 2：候选生成治理链

`M04 → M05 → M09-compose → M12 → M06 → M07`

把"生成"从单一步骤变成被治理的候选过程。

### 主链 3：检索复用链

`M04/M05 → M10-retrieve → M12/M06 → M07`

优先复用已有积累，而不是总从零开始。

### 主链 4：外部能力接合链

`M05 → M11-invoke → M12 → M06 → M07`

把"调用工具"从临时技巧，升级为可治理执行路径。

### 主链 5：能力创造与制度化链

`M05/M10/M01 → M13 → M12 → M06 → M01 → M05`

缺口发现 → 创造 → 核验 → 评估 → 沉淀 → 路由复用。使系统真正具备"自扩展"能力。

## 五、两条治理闭环

### 闭环 A：质量治理闭环

`执行层产物 → M12 核验 → M06 评估 → 回流相邻原子 → 再执行`

避免"输出即完成"。

### 闭环 B：能力演化闭环

`历史缺口 → M13 创造 → M12 核验 → M06 评估 → M01 沉淀 → M05 路由复用`

让系统真正成长，而不是每次临时应对。

## 六、元部门复制链

`模式积累/能力缺口/母体模板 → M13-create → M02-identity → M12-verify → M06-evaluate → M01-memory → M05-route`

元部门不是单一固定组织，而是：可派生、可特化、可分叉、可继承、可再纳入治理。

## 七、Claude Code 运行绑定

抽象原子在 Claude Code 环境中落到真实执行关系上：

| 原子 | 运行绑定规则 |
|------|------------|
| M04 | 标准/重量拆解**必须**通过 Agent 工具创建独立子任务执行器 |
| M05 | 路由应形成真实执行归属（Agent 实例），而非口头说"交给谁" |
| M06 | 评估**必须**由独立 Agent 完成，且与执行上下文隔离 |
| M08 | 并行应体现为单条消息内多个 Agent 执行实体 |
| M11 | 调用应是实际接合外部能力，而非概念性描述 |

**单实体执行仅限** M04 轻量模式（子任务 ≤ 2 且无依赖，文件 ≤ 3）。

## 九、Agent 创建机制

M13 具备**三级创造能力**：

| 创建类型 | 触发条件 | 门控要求 |
|---------|---------|---------|
| Skill | 能力缺口 + 2次验证通过 + I/O 契约明确 | 最小可复用条件 |
| Agent | 结构性缺口（≥3次同类需求）+ 需要独立身份/状态/进化 | 五道门控（缺口→Skill不足→层归属→正交性→人类确认）|
| Meta-Department | 领域特化需求 + 完整13原子实例化 | 谱系追踪 + 治理同步 + 独立演化 |

架构膨胀安全阀：原子总数超过 18 时触发架构审查。

## 十、设计原则

1. **不可拆分性**：13 元中任何一个若再拆分，将丧失独立语义完整性
2. **纯抽象性**：所有描述不含具象领域词汇，可复用于任意业务域
3. **自我进化**：每个元内建基于反馈的自适应机制
4. **正交性**：各元职责边界清晰，无功能重叠
5. **治理优先**：每个原子同时回答"我是什么、我不是什么、我如何被治理、我如何演化"
6. **可演化架构**：M13 具备创造新原子与新元部门的能力，使框架本身可生长

## 十一、统一术语约束

| 术语 | 归属原子 | 禁止混用 |
|------|---------|---------|
| Candidate / 候选 | M09 | 不要把 M10 检索结果叫 candidate |
| Existing Object / 既有对象 | M10 | 不要把能力也叫 existing object |
| Existing Capability / 既有能力 | M11 | 不要把可召回对象和可调用能力混为一个词 |
| Evaluate / 评估 | M06 | 质量裁决，不判断真实性 |
| Verify / 核验 | M12 | 成立性检查，不判断质量优劣 |
| Decompose / 拆解 | M04 | 只管"拆成什么"，不管"谁做""何时做" |
| Sequence / 编排 | M08 | 只管推进秩序，不管任务本体 |
| Synthesize / 综合 | M07 | 只整合结果，不吞并其他原子职责 |
| Created Capability / 新创能力 | M13 | 不要把单次输出升级叫 created capability |
