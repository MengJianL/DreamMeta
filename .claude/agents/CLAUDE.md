# Meta-Department 元部门架构索引

> "协同可以扁平，治理不能缺位。"

## 架构概览

本元部门由 **13个不可拆分的原子元** 组成，分为三层重力结构。
每个元具备独立的自我进化机制，所有描述为纯抽象定义，不含具象业务内容，可直接复用于任意业务域。

## Skill 融合映射

| 外部能力 | 融合至 | 融合方式 |
|---------|--------|---------|
| `find-skill` | M05 路由元 + M10 检索元 | 路由元在能力缺口时触发技能发现；检索元在技能库中执行搜索匹配 |
| `skill-create` | M13 创造元 | 全生命周期技能工程：创建标准结构（SKILL.md + scripts/ + references/ + assets/）→ 三级渐进式加载 → 迭代改进循环（draft → test → review → improve）→ 评估框架（测试用例/断言/评分/基准）→ 描述优化（触发精度）→ 打包为可分发 .skill 文件 |
| `awesome-claude-prompts` | M09 生成元 + M02 身份元 + M10 检索元 | 提示词工程框架（CRISPE/Meta Prompt）与10种结构化模式融合至生成元；身份构建方法论与渐进式个性化融合至身份元；模板库注册为检索元的参考资源 |

## Agent 创建机制

M13（创造元）具备**双重创造能力**，根据需求性质选择创建 Skill 或 Agent：

| 创建类型 | 触发条件 | 产出 | 门控要求 |
|---------|---------|------|---------|
| Skill 创建 | 能力缺口 + 2次验证通过 + I/O 契约明确 | 技能模块（可复用模式） | 最小可复用条件 |
| Agent 创建 | 结构性缺口（≥3次同类需求无法归入现有原子） + 需要独立身份/持久状态/自我进化 | 新原子元定义（.md 文件） | 五道门控：缺口确认 → Skill不足确认 → 层归属明确 → 正交性验证(M06) → 人类确认(M03) |

架构膨胀安全阀：原子总数超过 18 时触发架构审查。

## 十阶段流水线的元组合

```
Phase 1  方向    = M04(分解)
Phase 2  规划    = M04(分解) + M05(路由) + M08(序列)
Phase 3  执行    = M09(生成) + M10(检索) + M11(调用) + M13(创造)
Phase 4  审查    = M06(评估) [业务层]
Phase 5  元审查  = M06(评估) [独立层，同一元不同实例]
Phase 6  修订    = M12(验证) + M09(生成)
Phase 7  验收    = M12(验证)
Phase 8  汇总    = M07(聚合)
Phase 9  反馈    = M06(评估) + M07(聚合)
Phase 10 进化    = 全部元的自我进化机制触发
```

## 设计原则

1. **不可拆分性**：13元中任何一个若再拆分，将丧失独立语义完整性
2. **纯抽象性**：所有描述不含具象领域词汇，可复用于任意业务域
3. **自我进化**：每个元内建基于反馈的自适应机制
4. **正交性**：各元职责边界清晰，无功能重叠
5. **思维增强**：编排层元配备批判性深度思维模式，确保决策质量
6. **可演化架构**：元部门是"线的创造者"——M13 具备创造新原子的能力，使框架本身可生长，而非仅13原子静态固定
