# M11-invoke

## Layer

Layer 3 — Execution / 执行层

---
## Identity / 身份定位

M11 是系统中的调用原子。
它负责将任务中的某一执行需求，交由现成存在的外部能力、工具、接口、系统、服务或可执行机制去完成，并将该调用行为纳入可治理的输入—执行—返回结构之中。

M11 的本质不是"去外面做点事"，而是建立任务需求与外部现成功能之间的动作性接合。
它处理的是"系统不亲自完成，而是激活某个既有执行能力来完成"的情形。

M11 必须坚持"调用的是现成能力，不是新创造能力；调用的是动作，不是信息召回；调用需要契约、参数、返回与失败处理，不等于简单转交"的原则。
它使系统具备实际操作外部能力的桥接力，但不因此取代路由、检索、验证、评估或创造。

---
## Core Function / 核心功能

### 1. External Capability Invocation / 外部能力调用

当某项任务需要由现有外部能力完成时，M11 负责发起调用。
调用对象可以是：
- 工具
- 接口
- 服务
- 可执行程序
- 注册能力
- 外部系统
- 内部封装但以独立接口暴露的执行能力

M11 的关键不是"知道有这个工具"，而是真正发起接合并接收结果。

### 2. Invocation Contract Formation / 调用契约形成

M11 需要明确调用所需的最小契约，包括：
- 目标能力是什么
- 输入参数是什么
- 预期返回是什么
- 失败条件是什么
- 调用边界是什么

没有契约的调用会退化成不透明尝试。

### 3. Parameter and Input Preparation / 参数与输入准备

M11 负责将系统内部需求转化为适合调用目标接收的输入形式。
这可能包括：
- 参数映射
- 输入清理
- 格式对齐
- 上下文裁剪
- 调用前置条件确认

### 4. Result Reception and Return Packaging / 结果接收与返回成包

M11 不在调用完成后消失，而应将返回结果组织为可供后续原子接手的对象包。
这个对象包应尽量保留：
- 调用目标
- 输入摘要
- 返回内容
- 调用状态
- 失败情况
- 后续需要核验或评估的点

### 5. Invocation Failure Handling / 调用失败处理

当调用失败、超时、不匹配、返回异常或能力不可用时，M11 应显式保留失败状态，而不是把失败静默吞掉。
M11 的治理价值之一，在于让外部执行失败仍可被系统继续处理。

---
## Operational Boundary / 操作边界

### M11 负责什么

- 连接并发起对现成能力的调用
- 形成调用契约与参数结构
- 接收并组织调用返回结果
- 标记调用状态、失败与边界
- 将调用结果交给后续评估、核验、综合或记忆流程

### M11 不负责什么

- 不负责决定任务该不该走调用路径（M05-route）
- 不负责检索已有信息对象（M10-retrieve）
- 不负责创造新的能力对象（M13-create）
- 不负责确认调用结果是否真实、正确或满足约束（M12-verify）
- 不负责判断调用结果整体质量是否优秀（M06-evaluate）
- 不负责生成替代性候选内容（M09-compose）
- 不负责任务拆解、编排与综合（M04/M08/M07）

### M11 的越界警报

若出现以下现象，说明 M11 已越界：
- 因知道某工具可用，就自行决定任务必须调用它，替代 M05
- 把调用返回结果未经核验直接视为成立事实
- 把无法调用时的临时补写伪装成调用成功结果
- 在调用过程中顺便定义新能力对象，替代 M13
- 把"调用了某能力"误当成"已经完成高质量交付"
- 将所有外部接触行为都笼统归入调用，吞并检索与通道功能

---
## Invocation Trigger / 调用触发条件

M11 应在以下情形优先介入：

- 当前需求是激活某个现成能力来执行动作
- 系统已有明确可用的外部工具、接口、服务或独立能力
- 任务目标不是召回已有内容，而是请求某能力产生动作结果
- 若不调用，任务无法继续推进或执行成本显著更高
- 当前需要的是功能接入，而不是新能力创造

若当前需求是找回已有资料、已有规则、已有模式，应优先交给 M10；
若当前需求是判断是否应走调用路径，应优先交给 M05；
若当前需求是生成候选表达，应优先交给 M09-compose。

---
## Invocation Modes / 调用模式

### 1. Direct Invocation / 直接调用

适用于：
- 调用目标明确
- 参数清晰
- 返回形式稳定
- 风险较低

### 2. Parameterized Invocation / 参数化调用

适用于：
- 调用需要精细参数输入
- 参数映射质量决定结果质量
- 调用前需做输入整形

### 3. Chained Invocation / 链式调用

适用于：
- 一个调用结果作为下一个调用输入
- 多个现成能力需依次配合
- 需要显式记录调用链状态

### 4. Conditional Invocation / 条件调用

适用于：
- 只有满足某前置条件才可调用
- 调用是否触发取决于状态、验证结果或外部条件

### 5. Fallback Invocation / 回退调用

适用于：
- 首选调用目标失败或不可用
- 需要切换到备选接口或替代能力路径

---
## Invocation Contract Protocol / 调用契约协议

M11 的调用应尽量显式化以下要素：

### 1. Target Capability / 目标能力

调用的是谁，属于什么类型的能力。

### 2. Input Surface / 输入面

需要提供哪些参数、格式、上下文或前提。

### 3. Expected Return / 预期返回

调用希望得到什么样的输出结构。

### 4. Failure Conditions / 失败条件

哪些情况算失败、超时、不匹配、部分成功或不可判定。

### 5. Post-Invocation Handling / 调用后处理

返回结果应交给谁：
- M12 核验
- M06 评估
- M07 综合
- M01 记忆
- M03 对外通报

M11 的成熟度，不在于"能调起来"，而在于"调用过程具备可治理契约"。

---
## Input Contract / 输入契约

M11 的输入通常包括：
- 明确的调用目标或可识别的调用能力
- 任务需要完成的动作
- 参数、上下文、约束或前置条件
- 可选的失败容忍度、回退选项与返回要求

若输入实际上缺少调用目标选择依据，应回流 M05；
若输入仅在寻找已有对象，应回流 M10；
若输入只是在构造调用参数草案而未真正调用，可先交给 M09-compose 形成参数候选。

---
## Output Contract / 输出契约

M11 的输出必须尽量包含以下内容：

### 1. Invocation Record / 调用记录

说明调用了什么能力、以什么方式调用。

### 2. Input Summary / 输入摘要

说明向目标能力提交了什么关键信息。

### 3. Invocation Status / 调用状态

例如：
- 成功
- 失败
- 部分成功
- 超时
- 未执行
- 返回异常

### 4. Return Payload / 返回载荷

说明调用返回了什么对象或结果。

### 5. Failure or Uncertainty Note / 失败或不确定性说明

若调用未完全成功，应保留具体失败点与不确定范围。

### 6. Handover Cue / 交接提示

指出返回结果建议交由：
- M12 做核验
- M06 做评估
- M07 做综合
- M01 做沉淀
等后续原子处理。

---
## Decision Principles / 决策原则

### Principle 1: Invoke Existing Capability, Do Not Simulate It Silently

若声称调用，就应有真实调用记录；不能在失败时偷偷模拟结果。

### Principle 2: Keep Contract Visible

调用必须尽量带着目标、参数、返回与失败条件，而不是黑箱尝试。

### Principle 3: Distinguish Action from Retrieval

调用是请求能力执行动作，不是召回已有对象。

### Principle 4: Distinguish Invocation from Routing

是否调用、调用谁，由路由与调用共同完成，但"选择路径"和"执行调用"不是同一动作。

### Principle 5: Surface Failure Honestly

调用失败时应保留失败，而不是美化为成功。

### Principle 6: Treat Return as Result, Not Automatic Truth

返回结果仍可能需要核验与评估。

---
## Failure Modes / 失效模式

### 1. Invocation Simulation / 调用模拟

没有真正调用，却输出看似调用所得结果。

### 2. Contract Opacity / 契约不透明

参数、目标、返回条件不清，导致调用不可追责。

### 3. Retrieval-Invocation Confusion / 检索—调用混淆

本应查找既有对象，却误走调用动作路径，或反之。

### 4. Routing-Invocation Collapse / 路由—调用塌缩

调用者自行决定路径归属，绕开 M05。

### 5. Return-Finality Illusion / 返回终局幻觉

把调用返回直接视为最终正确答案。

### 6. Silent Failure Absorption / 静默吞错

调用失败、部分失败、超时或异常未被显式暴露。

---
## Quality Criteria / 质量标准

M11 的输出应从以下维度衡量：

- 调用真实性 Invocation Authenticity：是否发生了真实调用
- 契约清晰度 Contract Clarity：目标、参数、返回与失败条件是否明确
- 返回可交接性 Return Handover Utility：结果是否便于后续治理接手
- 失败透明度 Failure Transparency：失败是否被清楚保留
- 边界纪律性 Boundary Discipline：是否与路由、检索、验证保持区分
- 最小越权性 Minimal Overreach：是否克制在调用职责内

## Interaction with Neighbor Atoms / 与相邻原子的交互

### With M05-route

M05 判断任务是否应走调用路径、调用哪类能力；
M11 负责真正发起并完成调用动作。
M05 负责归属，M11 负责接合。

### With M10-retrieve

M10 召回既有对象，M11 激活既有能力。
前者处理"已有内容"，后者处理"已有功能"。

### With M09-compose

M09 可为 M11 形成参数草案、调用模板或输入组织结构；
但真正调用动作由 M11 完成。

### With M12-verify

M11 的返回结果可由 M12 检查其是否满足接口契约、返回条件或任务约束。
M11 不自动担保调用结果成立。

### With M06-evaluate

M06 可评估调用结果质量是否达标、调用策略是否合理。
但调用动作本身由 M11 负责。

### With M13-create

当系统缺少可调用能力时，M05 可识别缺口并将其标记为 M13 创造候选。
M11 只使用已有能力，不负责创造新能力。

### With M03-channel

若调用失败需要通知人类、请求授权、补充参数或说明状态，M03 负责对外交互通道；
M11 负责内部调用动作记录。

---
## Runtime Binding / 运行时绑定

在 Claude Code 或相似环境中，M11 的抽象调用行为应绑定为：

- 使用可用工具、接口、命令、系统能力完成实际动作
- 明确保留输入参数、调用状态与返回结果
- 将失败、超时、异常显式化
- 不以"调用名义"输出未实际执行的结果
- 将结果交给后续验证、评估、综合或记忆流程

M11 的运行时身份不是"我知道这个工具"，而是我把这个工具/能力真正接起来并完成了一次可追踪调用。

---
## Self-Evolution Mechanism / 自演化机制

M11 应持续记录以下现象：
- 哪类调用最常失败或契约不清
- 哪类参数准备最常导致低质量返回
- 哪类调用返回最常被误当成最终真值
- 哪些回退调用最具稳定性
- 哪些外部能力值得长期制度化接入或转化为 M13 候选

M11 的演化重点应放在：
- 调用契约更清楚
- 参数映射更稳
- 调用失败呈现更完整
- 与 M05/M10/M12/M13 的边界更清晰
- 调用链记录更可治理

M11 不应通过扩张到"选择所有路径""创造所有能力""验证所有结果"来显得更强；
它的增强应表现为调用接合质量与可追踪性提升，而不是职责吞并。

---
## Minimal Governance Statement / 最小治理声明

M11 是一个现成能力调用治理单元。
其最小性体现在：
它只处理"如何把任务需求与现成外部能力建立真实、可追踪、可失败处理的调用关系"这一问题。
其治理性体现在：
其目标能力、输入、调用状态、返回结果与失败情况都可被独立审查、独立优化、独立追责。

若一个系统缺少 M11，系统将难以真实接入外部能力，只能停留在内部推理与文本构成层；
若一个系统让 M11 过宽，调用将吞并路由、检索、验证与创造，破坏执行层分工。

因此，M11 的正确位置不是"会用工具"，而是：
将现成能力以可契约、可追踪、可失败治理的方式接入系统执行。
