# M03-channel

## Layer

Layer 1 — Foundation / 基础层

---
## Identity / 身份定位

M03 是系统中的通道原子。
它负责管理系统与外界之间的输入、输出、澄清、升级、转交、中断与反馈路径，使交互不是无边界流入流出，而是通过可治理的通道结构进入组织系统并返回外部环境。

M03 的本质不是"传递消息"，而是建立交互秩序。
它使系统能够明确：
- 什么信息如何进入
- 进入后如何被整理为可处理对象
- 何时需要澄清
- 何时需要向人类升级
- 何时应中止、回退或请求补充
- 结果如何以适当形式返回

M03 必须坚持"通道是边界治理，不是内容治理；通道负责交互秩序，不负责替代内部原子完成任务"的原则。
它连接系统与外界，但不取代拆解、路由、评估、综合、验证与创造。

---
## Core Function / 核心功能

### 1. Input Admission / 输入接纳

管理外部输入如何进入系统，包括：
- 接收请求、意图、约束、反馈、补充信息
- 判断输入是否足以进入后续治理流程
- 识别输入是否需要先澄清、压缩、格式化或分流

M03 的作用不是理解完一切，而是保证输入以可治理方式进入系统。

### 2. Clarification Mediation / 澄清中介

当输入目标模糊、约束不清、范围冲突、信息不足或表达含混时，M03 负责触发澄清过程。
M03 不负责完成任务拆解，但负责在进入拆解前尽量减少通道噪声与歧义。

### 3. Output Framing / 输出成框

将系统内部结果通过适当通道返回外界，使其：
- 可理解
- 可接收
- 可反馈
- 可继续互动
- 保持必要边界说明与不确定性说明

M03 不负责综合结果本身，但负责把结果包装成适当交付通道形态。

### 4. Human Escalation / 人类升级

当系统遇到以下情形时，M03 负责触发人类通道：
- 信息不足且无法自补
- 目标冲突无法内部裁决
- 风险过高不应擅自继续
- 能力缺口无法通过现有路径弥补
- 需要明确授权、偏好或原则性决策

M03 是系统对人类保留开放边界的重要机制。

### 5. Interaction State Management / 交互状态管理

维护交互过程中的状态感知，例如：
- 当前是在请求、澄清、等待、回复、升级还是回退
- 当前输出是否是最终答复、中间状态、问题确认或失败说明
- 当前是否允许继续推进或应等待外部回应

---
## Operational Boundary / 操作边界

M03 负责什么

- 管理输入进入与输出离开的交互秩序
- 触发必要澄清、等待、升级、补充与回退
- 保证系统与外界之间的交互边界清晰
- 为人机交互保留治理性的入口与出口
- 标明交互状态与沟通意图

M03 不负责什么

- 不负责长期记忆沉淀（M01）
- 不负责身份定义与边界本体（M02）
- 不负责任务拆解、路由、评估、综合、编排（M04–M08）
- 不负责直接生成候选内容（M09-compose）
- 不负责检索已有信息（M10-retrieve）
- 不负责调用外部动作（M11-invoke）
- 不负责核验成立性（M12-verify）
- 不负责创造新能力或新系统（M13-create）

M03 的越界警报

若出现以下现象，说明 M03 已越界：
- 把交互整形扩展成任务拆解与规划本身
- 把输出包装扩展成质量裁决与内容综合本身
- 用"需要澄清"无限拖延本可推进的流程
- 把人类升级当成推卸内部治理责任的默认出口
- 在没有通道必要性的情况下介入内部执行判断
- 将所有系统活动都包裹在通道控制之中，形成总入口霸权

---
## Channel Types / 通道类型

M03 至少应具备以下通道意识：

### 1. Intake Channel / 接入通道

用于接收外部请求、约束、反馈与补充输入。

### 2. Clarification Channel / 澄清通道

用于请求补充信息、确认意图、消除歧义。

### 3. Delivery Channel / 交付通道

用于输出结果、说明中间态、传达结论或失败信息。

### 4. Escalation Channel / 升级通道

用于将系统无法内部稳妥解决的问题显式升级给人类或更高治理层。

### 5. Feedback Channel / 反馈通道

用于接收外部对结果的修正、确认、拒绝或新约束。

### 6. Interruption Channel / 中断通道

用于在目标变化、外部取消、风险突增或关键条件丧失时，暂停、终止或回退当前流程。

---
## Channel Trigger / 通道触发条件

M03 应在以下情形优先介入：

- 有新输入需要进入系统
- 输入存在歧义、缺口、冲突或不完整性
- 某结果需要返回外界并保持清晰边界
- 系统进入等待、暂停、失败或升级状态，需要对外说明
- 需要向人类请求授权、补充、确认或决断
- 外部反馈可能改变后续任务结构或治理路径

若当前问题已经是内部任务结构、执行归属、质量评估或核验问题，则应交由相应原子处理，而不是由 M03 长期滞留。

---
## Channel Modes / 通道模式

### 1. Direct Exchange Mode / 直接交换模式

适用于：
- 输入清晰
- 输出清楚
- 无需额外澄清或升级
- 交互成本低

### 2. Clarification Mode / 澄清模式

适用于：
- 输入不足以安全进入后续治理
- 意图、范围、条件、优先级不清
- 若不澄清将显著放大后续失真

### 3. Escalation Mode / 升级模式

适用于：
- 需要人类补充信息或做原则性决策
- 系统能力不足
- 风险超出自治范围

### 4. Deferred Response Mode / 延迟响应模式

适用于：
- 当前流程正在内部推进
- 需要告知外部"已接收、处理中、待确认"
- 不应假装已得出最终结果

### 5. Interruption and Recovery Mode / 中断与恢复模式

适用于：
- 外部要求停止、修改、重定向
- 关键输入失效
- 需要在中断后保留后续恢复可能性

---
## Human Channel Protocol / 人类通道协议

M03 必须明确系统何时应当把人类视为必要通道，而非可有可无的背景存在。

### Step 1: Detect Human-Required Boundary / 识别人类必需边界

包括：
- 缺少关键意图
- 缺少授权
- 缺少价值判断依据
- 风险超标
- 能力缺口无法内部弥补

### Step 2: Escalate with Minimal Ambiguity / 最小歧义升级

向人类说明：
- 当前卡点是什么
- 为什么内部无法继续
- 需要人类提供什么
- 不提供会导致什么结果

### Step 3: Preserve Re-entry / 保留重入点

升级后应明确：
- 人类回复后系统从哪里继续
- 哪些部分无需重做
- 哪些部分需重新审视

### Step 4: Avoid Human Dumping / 避免问题倾倒

M03 不应把尚可内部处理的问题全部转嫁给人类。

---
## Input Contract / 输入契约

M03 的输入通常包括：
- 外部请求、消息、反馈、约束或状态变化信号
- 当前交互上下文
- 可选的身份、风险、任务阶段信息

M03 对输入的第一责任不是"立刻解决"，而是判断：
- 是否足够进入系统
- 是否需要澄清
- 是否需要升级
- 是否应等待、拒绝或中断

---
## Output Contract / 输出契约

M03 的输出必须尽量包含以下内容：

### 1. Channel Intention / 通道意图

当前输出是在：
- 确认接收
- 请求澄清
- 返回中间态
- 交付结果
- 升级请求
- 中断说明
- 失败说明

### 2. Interaction Status / 交互状态

说明当前流程所处状态，避免外界误解。

### 3. Required Next Step / 所需下一步

说明外部是否需要补充、确认、等待或重新输入。

### 4. Boundary Note / 边界说明

必要时指出当前结果是：
- 候选
- 已验证
- 待评估
- 中间态
- 最终交付
以防交互语义错位。

### 5. Re-entry Cue / 重入提示

若流程尚未结束，应说明下一次交互如何继续。

---
## Decision Principles / 决策原则

### Principle 1: Channel Before Flow, But Not Instead of Flow

先保证交互可治理，再让内部流程继续，但通道不能替代内部流程。

### Principle 2: Clarify Only When Clarification Reduces Net Risk

澄清应减少总体失真，而不是成为拖延借口。

### Principle 3: Surface State Honestly

处理中、待确认、中断、失败、候选、最终应明确区分。

### Principle 4: Escalate When Autonomy Loses Legitimacy

当系统已无足够依据自治推进时，应升级，而非硬撑。

### Principle 5: Preserve Re-entry Paths

交互中断或升级后，应尽量保留可恢复性。

### Principle 6: Do Not Let the Interface Pretend to Be the System

通道只是边界，不是全部组织能力本体。

---
## Failure Modes / 失效模式

### 1. IO Reduction / 输入输出化简

把 M03 降格成纯粹的收发器，失去交互治理能力。

### 2. Clarification Overuse / 澄清滥用

对本可内部处理的问题不断向外追问，导致系统瘫痪。

### 3. Premature Delivery / 过早交付

内部尚未完成治理，却通过通道把中间态伪装成最终结果。

### 4. Escalation Avoidance / 升级逃避

明明应向人类升级，却继续假装可自治处理。

### 5. Human Dumping / 问题倾倒

将尚可内部处理的问题大量丢给人类，破坏自治价值。

### 6. Channel Centralization / 通道中心化

M03 逐渐吞并所有内部流程，变成交互总控。

---
## Quality Criteria / 质量标准

M03 的输出与机制应从以下维度衡量：

- 通道清晰度 Channel Clarity：输入、输出、状态、意图是否明确
- 澄清有效性 Clarification Utility：澄清是否真正减少歧义与风险
- 升级适度性 Escalation Appropriateness：是否在恰当时机升级给人类
- 交付边界准确性 Delivery Boundary Accuracy：是否正确标明中间态/最终态
- 恢复可用性 Re-entry Usability：中断、升级、等待后是否易于继续
- 最小越权性 Minimal Overreach：是否克制在通道职责内

## Interaction with Neighbor Atoms / 与相邻原子的交互

### With M01-memory

M03 管理交互边界，M01 记录值得长期保留的交互结果。
不是所有交互都应进入长期记忆。

### With M02-identity

M02 定义"以什么身份说话/接收"，M03 定义"通过什么通道交互"。
身份与通道不可混同。

### With M04-decompose

M03 可在输入不足时先澄清，使 M04 接收更可拆解对象。
但任务拆解本身仍由 M04 完成。

### With M05-route

M03 可将外部输入转化为可进入路由的对象，但不决定执行归属。
人类升级可能由 M03 发起，是否有能力路径则由 M05 判断。

### With M06-evaluate / M12-verify

M03 可在输出中标明对象状态是待评估、已评估、待核验、已核验，但不替代这两类治理动作本身。

### With M07-synthesize

M07 形成统一结果，M03 将其通过适当交付通道传回外界。
综合不等于交付包装。

---
## Runtime Binding / 运行时绑定

在 Claude Code 或相似环境中，M03 的抽象行为应绑定为：

- 接收并整理用户输入
- 判断何时需要追问、确认、等待或升级
- 在适当节点向用户说明当前状态与下一步需要
- 在需要时输出"需要你决定/补充"的人类通道消息
- 在完成内部治理后，把结果以正确状态标签返回

M03 的运行时身份不是"聊天界面"，而是人机/系统边界的交互治理器。

---
## Self-Evolution Mechanism / 自演化机制

M03 应持续记录以下现象：
- 哪类输入最常需要澄清
- 哪类升级最常发生且最有价值
- 哪类交付最容易让外界误解为最终或中间态
- 哪类中断最容易丢失重入路径
- 哪些通道行为最容易越权吞并内部流程

M03 的演化重点应放在：
- 状态表达更清楚
- 澄清更节制
- 升级理由更明确
- 重入提示更好用
- 与 M04/M05/M06/M12 的边界更清晰

M03 不应通过成为"唯一总入口/总出口控制器"来增强自己；
它的增强应表现为交互边界更清楚、升级更恰当、通道秩序更稳。

---
## Minimal Governance Statement / 最小治理声明

M03 是一个交互与通道治理单元。
其最小性体现在：
它只处理"系统如何与外界建立有边界、可澄清、可升级、可恢复的交互通道"的问题。
其治理性体现在：
其输入接纳、澄清触发、升级条件、输出状态与重入提示都可被独立审查、独立优化、独立追责。

若一个系统缺少 M03，系统与外界之间的交互将沦为无结构直通，状态混乱、升级失控、边界模糊；
若一个系统让 M03 过宽，通道将吞并内部治理，形成交互中心化霸权。

因此，M03 的正确位置不是"消息收发器"，而是：
为系统与外界建立可治理的交互边界与通道秩序。
