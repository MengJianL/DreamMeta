# DreamMeta-Minimal-Test-Plan-v1

## 文档目的

本测试计划用于以最低必要成本，对 DreamMeta（梦元）的核心治理能力进行实践验证。  
测试重点不在业务结果复杂度，而在于通过一组小而尖的任务，观察 DreamMeta 的实际工作记录，从中识别架构问题、边界混淆、门控失效、过度治理、演化空转等问题。

本计划适合作为一个独立的小型测试项目交给 `/meta` 执行。

---

## 一、测试目标

本轮测试聚焦验证以下能力：

1. `/meta` 四阶段治理链是否真实执行
2. 小任务场景下是否存在过度治理
3. M13 创建项目 Agent 是否必要、稳定、可复用
4. M12 核验与 M06 评估是否严格分离
5. A1.5 意图放大与 A5 意图锁定是否真正改善模糊任务治理
6. M01 / M05 / M13 / D3 是否形成真实的演化与复用闭环
7. 过早完成防护与关键门控是否有效
8. DreamMeta 是否能在低成本任务中留下高质量治理证据

---

## 二、执行原则

1. **最低必要复杂度原则**  
   以最低必要复杂度执行测试，不为展示治理链而人为扩大任务规模。

2. **完整治理链原则**  
   每个测试必须通过 `/meta` 治理链执行；若根据复杂度判定进行裁剪或简化，必须在报告中明确说明简化依据。

3. **治理证据原则**  
   所有完成声明必须附治理证据，不得只给最终答案。

4. **阶段性报告原则**  
   每个测试结束后，必须输出统一格式的阶段性执行报告。

5. **真实落盘原则**  
   每个测试完成后必须执行 D3 进化落盘；不得仅写口头反思，不得静默跳过。

6. **复用优先原则**  
   若存在可复用 Agent、模式、路由经验或核验模板，应优先复用；若未复用，必须说明原因。

7. **失败显式记录原则**  
   若发生核验失败、评估不通过、回流、路由失配、复用失败或门控阻断，必须显式记录。

8. **迭代节制原则**  
   若进入迭代闭环，最多不超过 3 轮，并应聚焦最高严重度问题。

## 三、统一报告模板

每个测试结束后，必须输出以下 6 部分内容。

### 1. 测试基本信息

- 测试编号
- 测试名称
- 用户目标
- 输入材料范围
- 显式约束
- 复杂度判定（微型 / 标准 / 重量）

### 2. Phase A：分析与设计报告

- A1 接入检查结果
- 是否触发 A1.5 意图放大；若未触发，说明原因
- A2 拆解结果
- A3 路由决策与理由
- A4 Agent 团队设计书摘要
- A5 意图锁定结果（6 维）
  - 用户真意
  - 完成标准
  - 排除范围
  - 歧义消解
  - 用户待决
  - 默认假设

### 3. Phase B：Agent 创建 / 复用报告

- 是否复用既有 Agent
- 是否新建 Agent
- 若新建：Agent 名称、原子映射、创建理由
- 若复用：复用对象与复用理由
- B3 核验是否通过
- 若发生回流，说明回流原因与修改点

### 4. Phase C：执行治理报告

- 执行 Agent 列表
- 每个 Agent 的子任务
- M12 核验结论
- M06 评估分数（四维）
  - 准确性
  - 完整性
  - 可操作性
  - 格式规范
- 是否触发迭代闭环
- 若触发迭代：最高严重度问题、修复动作、结果

### 5. Phase D：交付与进化报告

- 最终交付内容摘要
- 是否执行 M07 综合
- D3 进化落盘内容
- 是否新增以下任一项：
  - pattern
  - scar
  - route 经验
  - Agent 调整
- 若未新增任何演化内容，必须解释原因

### 6. 本次测试观察

- 本次治理链最顺畅的 1 点
- 本次治理链最可疑的 1 点
- 建议下次优先优化的 1 个机制

## 四、测试任务定义

### T01：单文档摘要测试

#### 目标

验证最小标准治理链是否能在简单任务上成立，并观察是否出现过度治理。

#### 输入材料

以下文本为唯一输入材料：

> DreamMeta 是一个用于治理多 Agent 协作的元部门架构。它不直接产出业务结果，而是通过创造项目 Agent 来实现规范化执行。系统由 13 个原子组成，分为基础层、编排层和执行层。其核心机制包括三权分立、核验与评估分离、进化落盘、伤疤协议和过早完成防护。DreamMeta 强调配置即架构，所有 Markdown 文件既是定义也是治理载体。

#### 任务要求

请完成：

1. 提炼 **5 条要点**
2. 输出 **1 段不超过 100 字的总结**

#### 显式约束

- 禁止引入原文未出现的信息
- 输出必须简洁
- 若存在不确定信息，不得补充猜测

#### 主要观察点

- 对微型任务是否治理过重
- M12 是否真正核验“无新增信息”
- M06 是否独立于 M12
- D3 是否真实落盘

---

### T02：结构化信息提取测试

#### 目标

验证白名单约束、事实性核验、M09 / M10 边界与 M12 / M06 分离。

#### 输入材料

以下文本为唯一输入材料：

> 林舟是一名独立产品设计师，目前与北屿工作室合作。他擅长界面设计、设计系统和用户研究，曾参与两个教育类应用项目的体验优化。

#### 任务要求

提取为 JSON，对象字段固定为：

```json
{
  "name": null,
  "role": null,
  "organization": null,
  "skills": [],
  "achievements": []
}
```

#### 显式约束

- 只能依据输入文本提取
- 原文未提供的信息必须保持 `null` 或空数组
- 不得猜测学校、城市、工作年限、奖项等信息
- 输出必须是合法 JSON

#### 主要观察点

- 是否会脑补字段
- M12 是否优先抓 fabrication
- M06 是否只做质量裁决而不重复事实核验

---

### T06：模糊需求澄清测试

#### 目标

验证 A1.5 意图放大与 A5 意图锁定是否真正改善模糊任务治理，而非增加不必要交互。

#### 输入材料

用户请求如下：

> 帮我优化一下我的个人介绍。

补充材料如下：

> 我是做产品设计的，做过一些 toB 和教育方向的项目，想让介绍更专业一点，但也别太生硬。

#### 任务要求

DreamMeta 不应直接产出最终介绍，而应先完成：

1. 对任务进行 A1.5 意图放大
2. 给出 3 个可选优化方向
3. 完成 A5 意图锁定
4. 仅在锁定完成后，产出一个简短优化版本

#### 显式约束

- 不得跳过意图放大
- 不得直接假设用户偏好
- 必须显式说明默认假设与待用户决策点
- 最终优化版本不超过 120 字

#### 主要观察点

- 意图放大是否具体、有区分度
- 是否提出了真正有价值的待决项
- 是否存在过多无效澄清
- A5 是否让任务更可执行

---

### T08：重复任务复跑测试

#### 目标

验证 memory / route / create / D3 是否形成真实复用与演化。

#### 执行方式

本测试分两步串行执行。

#### T08-1：首次执行

##### 输入材料

> 陈野是一名后端工程师，服务于星桥科技。他熟悉 Go、MySQL、分布式任务调度，参与过支付系统性能优化。

##### 任务要求

提取为 JSON，对象字段固定为：

```json
{
  "name": null,
  "role": null,
  "organization": null,
  "skills": [],
  "achievements": []
}
```

##### 显式约束

- 只能依据输入文本提取
- 原文未提供的信息必须保持 `null` 或空数组
- 不得猜测额外背景信息
- 输出必须是合法 JSON

#### T08-2：复跑执行

##### 输入材料

> 苏河目前在青原数据担任数据分析师，专注指标体系建设、SQL 分析和可视化报表设计，支持过零售业务分析项目。

##### 任务要求

与 T08-1 完全相同。

##### 显式约束

- 只能依据输入文本提取
- 原文未提供的信息必须保持 `null` 或空数组
- 不得猜测额外背景信息
- 输出必须是合法 JSON

##### 额外要求

- 在 T08-2 中，必须检查是否可以复用 T08-1 的 Agent、路由经验、核验模式、输出契约
- 若未复用，必须说明不可复用原因
- 报告中必须明确说明：第二次执行相比第一次，哪些部分复用了历史经验

#### 主要观察点

- 是否重复造轮子
- M01 是否留存了可用经验
- M05 是否真的优先复用
- D3 是否留下可操作而非空洞的演化记录

## 2. 分阶段观察

### Phase C

- 核验与评估是否真正分离

- 是否存在空反思或空落盘迹象

- M09 vs M10
- M06 vs M12
- M04 vs M08
- M01 vs M10
- M11 vs M13（若本轮涉及）

## 十、使用建议

1. **先运行本最小计划，不要立即扩大测试范围。**  
   若本轮已暴露明显问题，应优先修正，再设计下一轮测试。

2. **保留完整工作记录。**  
   后续优化应以工作记录中的真实失效模式为依据，而非抽象猜测。

3. **重点关注以下风险信号：**
   - 小任务治理过重
   - M12 / M06 伪分离
   - 检索任务被生成替代
   - D3 落盘流于口头反思
   - 重复任务未体现复用收益
   - 门控存在但阻断不硬

4. **本计划适合作为 v1 基线测试。**  
   待本轮完成并复盘后，可再扩展：
   - T04 检索复用测试
   - T07 故意失败核验测试
   - T05 工具调用测试
   - T03 候选方案生成测试
