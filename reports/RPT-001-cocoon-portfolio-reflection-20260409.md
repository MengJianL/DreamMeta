## 二、三个版本的失败链路

### v1：单Agent + 无Skill调用
```
用户需求 → 单个Agent一把梭 → 布局粗糙 → 用户否决
                ↑
        根因：未拆解为专业分工
```
**现象**: "做的跟垃圾没什么区别，Agent就一个"
**根因**: M04-decompose 未被激活，任务未拆解为设计/内容/实现三个维度

### v2：多Agent但约束空洞
```
用户需求 → 3个Agent → 设计规范无视觉锚 → 头像144px → 用户否决
                      → 内容Agent编造经历 → 用户否决
```
**现象**: 头像巨大、虚构时间线和项目
**根因**: Agent定义中的约束停留在原则声明层（"不得编造"），缺乏可执行的强制机制

### v3：严格约束 + 抽象命名
```
用户需求 → 3个M09-Agent → ASCII mockup强制约束 → 通过
                        → 严格内容白名单 → 通过
```
**修复**: 约束从"原则声明"升级为"可验证的具体规则"

## 三、伤疤根因分析

### scar-001: Skill 名称路由错配

| 维度 | 分析 |
|------|------|
| 表层原因 | 用户说"frontend"，实际Skill名"frontend-slides" |
| 深层原因 | **M05-route 缺乏 Skill 名称验证环节** |
| 治理缺陷 | M05 路由设计中没有强制要求在分配 Skill 前先用 `find-skills` 确认实际名称 |
| 改进方向 | M05 原子定义应增加：路由到 Skill 时，必须先经过 M10-retrieve 确认 Skill 实际注册名 |

### scar-002: 设计输出缺乏视觉锚点

| 维度 | 分析 |
|------|------|
| 表层原因 | 设计Agent只输出文字数值，工程师自行发挥 |
| 深层原因 | **M09-compose 输出契约未定义"可视化约束"这一维度** |
| 治理缺陷 | M12-verify 核验时缺少"设计-实现一致性"检查项 |
| 改进方向 | M09 原子的输出契约应区分"数值规范"和"视觉锚点"两种产出类型，涉及视觉产出时 ASCII mockup 为必须项 |

### scar-003: 内容虚构（最严重）

| 维度 | 分析 |
|------|------|
| 表层原因 | Content Agent 编造了用户未提供的经历和项目 |
| 深层原因 | **"不得编造"是否定式原则声明，LLM 天然倾向补全而非留空** |
| 治理缺陷 | M12-verify 核验清单未包含"事实性核验"（fabrication check） |
| 改进方向 | 1. Agent 定义中约束必须用**白名单模式**（"只允许X"）而非黑名单模式（"不要Y"）；2. M12 核验必须增加 fabrication check 作为标准检查项 |

## 四、系统性治理缺陷

### 缺陷 1: 约束声明 ≠ 约束执行

**现象**: Agent定义中写了"不得编造"、"头像不得超过96px"，但执行Agent完全无视。

**根因**: 原则性声明（"请不要X"）对 LLM 几乎无约束力。LLM 的默认行为是补全和发挥，而非克制和留空。

**改进方向**:
- 约束必须是**白名单模式**："Experience 栏目只允许写入以下内容：[枚举]"
- 约束必须是**可验证的**：M12 核验时能逐项对照检查
- 约束必须有**具体数值**："头像 96px" 而非 "头像不要太大"

### 缺陷 2: M12-verify 核验清单不完整

**现象**: v2 通过了核验但仍包含虚构内容和超大头像。

**根因**: M12 核验时只检查了"文案是否完整"、"构建是否通过"，未检查：
- 事实性（fabrication check）
- 设计-实现一致性（mockup compliance）
- 用户约束遵从性（constraint compliance）

**改进方向**: M12 原子定义应增加三类标准核验维度：
1. **事实性核验** — 所有内容是否有用户输入依据
2. **规范一致性核验** — 实现是否严格匹配设计规范数值
3. **约束遵从性核验** — 用户明确的禁止项是否被违反

### 缺陷 3: Agent 命名过于具体，违反抽象哲学

**现象**: "Content-Writer-Agent"、"Frontend-Engineer-Agent" 等命名包含领域特定词汇。

**根因**: 元部门的设计原则是 "pure-abstract, domain-agnostic"，但在创建项目 Agent 时，M05-route 自然倾向于使用具体领域词汇命名，因为具体名称更容易理解。

**改进方向**: 
- 项目 Agent 命名应遵循 `M##-[抽象角色]` 模式（如 M09-Content、M09-Structure、M09-Implement）
- M05 路由时应基于原子映射命名，而非基于人类职业标签
- 抽象命名 ≠ 含糊命名：名称应体现功能本质而非领域身份

### 缺陷 4: 三权分立执行不完整

**现象**: v2 的"核验通过 20/20"实际上是执行Agent自我评估，并非独立 M12 核验。

**根因**: 在执行链路中，M12-verify 和 M06-evaluate 未被真正派遣为独立 Agent 实例。核验和评估被内嵌在执行Agent的同一上下文中，违反了三权分立原则。

**改进方向**: 
- `/meta` 流程必须强制派遣独立的 M12 Agent 和 M06 Agent
- 核验 Agent 不得接收执行 Agent 的上下文，只接收产物本身
- 执行状态中的"质量评估"必须来自独立 M06 Agent，不可自评

## 六、改进建议清单（供梦元总部）

### 原子层面改进

| 优先级 | 原子 | 改进项 |
|--------|------|--------|
| P0 | M09-compose | 输出契约增加"视觉锚点"维度，涉及视觉产出时 ASCII mockup 为必须项 |
| P0 | M12-verify | 核验清单增加三类标准维度：事实性、规范一致性、约束遵从性 |
| P0 | M05-route | 路由到 Skill 时必须先经 M10 确认实际名称；Agent 命名遵循原子映射模式 |
| P1 | M04-decompose | 拆解产出中标注每个子任务的约束类型（白名单/数值/格式） |
| P1 | M06-evaluate | 评估维度增加"用户约束遵从度"（0-5分） |
| P2 | M08-sequence | 编排设计中增加强制门检：设计→实现之间必须有 M12 核验设计产出的完整性 |

### 治理机制改进

| 优先级 | 机制 | 改进项 |
|--------|------|--------|
| P0 | 约束表达 | 所有约束从黑名单模式（"禁止X"）改为白名单模式（"只允许Y"） |
| P0 | 三权分立 | `/meta` 流程强制派遣独立 M12 和 M06 Agent 实例，禁止自评 |
| P1 | fabrication check | 作为 M12 标准核验项，任何涉及用户事实信息的产出必须经过 |
| P1 | Agent 命名规范 | 项目 Agent 统一使用 `M##-[抽象角色]-Agent` 格式 |
| P2 | 伤疤回流 | 每个伤疤的 prevention_rule 应自动注入相关原子定义的 Failure Modes 章节 |

## 七、本项目伤疤的共同教训

> **原则性声明对 LLM 没有约束力。**
> 
> "不要编造" ≈ "不要想粉色大象"。
> 
> 有效的约束是：**白名单枚举 + 具体数值 + 独立核验**。

这是本项目三个伤疤的统一根因。每一次失败都源于同一个模式：用原则性声明代替了可执行的具体约束。
